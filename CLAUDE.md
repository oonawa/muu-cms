# CLAUDE.md

実装中に判断に迷ったときはここに立ち返ること。

## ドキュメント

| ファイル | 内容 |
|----------|------|
| `docs/requirements.md` | 要件定義 |
| `docs/design/database.md` | DB設計・テーブル定義 |
| `docs/design/api.md` | API設計・エンドポイント仕様 |
| `docs/design/frontend.md` | フロントエンド実装指針（Async React） |
| `TODO.md` | 現在のタスクリスト |

## 開発フロー

`/plan` スキルで TODO.md を作成 → ユーザー承認 → TDD で実装。

```
1. /plan でタスクを TODO.md に書き出す
2. ユーザーが承認したら次へ
3. 正常系テストを1つ書く
4. テストが落ちることを確認（Red）
5. テストが通るように実装（Green）
6. 全件テストが通ることを確認
7. 2に戻る
```

## テストの書き方

- テスト名は**日本語**で、仕様として1行で表す
- 例: `ルームを作成できる`、`パラメータのnameは変更できない`

## 実装上の制約

### DB
- EloquentとLaravelのmigrationのみ。Raw SQLを書かない
- DB固有の関数（`JSON_EXTRACT` 等）を使わない
- JSONカラムへのアクセスは `$casts` 経由に統一する

### ドキュメントとの整合性
- 実装内容がドキュメントと食い違う場合は実装を止めてオーナーに確認する
- ドキュメントに記載のない仕様を勝手に実装しない
- ドキュメントを更新した場合は更新日を修正する
- `docs/` 以下に未決定と明記されている事項は着手前にオーナーに確認する

## GitHub 運用

| ツール | 役割 |
|--------|------|
| Issue | 計画の記録（何をやるか・なぜやるか） |
| TODO.md | 作業中のフロー管理 |
| PR | 作業の記録（実際にやったこと・差分） |

### フロー
```
1. Issue を作成する
2. ブランチを切る: feat/#<issue番号> / fix/#<issue番号>
3. TODO.md で実装を進める
4. PR を作成する（Closes #<issue番号>）
5. オーナーが確認 → マージ
6. git switch main && git pull origin main
```

### コミットメッセージ

prefix: `feat:` / `fix:` / `ref:` / `test:`

- 最終的な状態のみ記述（途中経過は書かない）
- うまくいかなかった点・別Issueにした事項は必ず記載
- 機能単位・関心事単位でコミットを分ける

### やらないこと
- オーナーの承認なしに push / PR作成 / マージをしない
- Issue を勝手にクローズしない
