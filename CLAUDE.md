# CLAUDE.md

このドキュメントはClaude Codeが作業を進めるための指針。実装中に判断に迷ったときはここに立ち返ること。

---

## ドキュメント構成

| ファイル | 内容 |
|----------|------|
| `docs/requirements.md` | 要件定義。何を作るかの全体像 |
| `docs/design/database.md` | DB設計。テーブル定義・設計原則 |
| `docs/design/api.md` | API設計。エンドポイント・レスポンス仕様 |
| `docs/design/frontend.md` | フロントエンド実装指針。デザイン・Async React |
| `TODO.md` | 現在のタスクリスト（作業開始時に作成・更新する） |

---

## 開発フロー

以下のサイクルを厳守すること。

```
1. TODO.md にタスクを書き出す
2. タスクを1つ選ぶ
3. 実装計画を記載する
    a. 予定作業の目的 / 内容（絶対にコードを書かず、自然言語のみで書く）
    b. 編集 or 追加の対象となるファイルまでのパス
    c. 設計変更を含む場合は、変更前後のディレクトリツリー構造も書く
4. TODO.md に対してユーザーへレビューを依頼する
5. ユーザーが承認したら次のステップへ進む
6. そのタスクの正常系テストを1つだけ書く
7. テストを実行して落ちることを確認する（Red）
8. テストが通るように実装する（Green）
9. 全件テストが通ることを確認する
10. 2に戻る
```

---

## テストの書き方

- テストケースは**日本語**で記述する
- テスト名はそのテストが何を確認するかを1行で表す仕様として書く
- テスト名の例：`ルームを作成できる`、`パラメータのnameは変更できない`

---

## 実装上の制約

### DB

- EloquentとLaravelのmigrationのみで実装する
- Raw SQLを書かない
- DB固有の関数（`JSON_EXTRACT` 等）を使わない
- JSONカラムへのアクセスは `$casts` 経由に統一する

### ドキュメントとの整合性

- 実装内容がドキュメントと食い違う場合は、実装を止めてオーナーに確認する
- ドキュメントに記載のない仕様を勝手に実装しない
- ドキュメントを更新した場合は更新日を修正する

### 未決定事項

`docs/` 以下のドキュメントに未決定と明記されている事項は、実装着手前にオーナーに確認すること。

## TODO.md の運用

作業開始時に `TODO.md` を作成し、タスクをリストアップする。完了したタスクはチェックを入れる。タスクが増えた場合はその都度追記する。

```markdown
# TODO

- [x] 完了したタスク
- [ ] 未着手のタスク
```

---

## レビュー担当

実装・テストはClaude Codeが行う。オーナーはドキュメントレビューと動作確認のみを担当する。

## GitHub 運用

GitHub はプロジェクトの記録ツールとして使う。

| ツール | 役割 | 内容 |
|--------|------|------|
| Issue | 計画の記録 | 何をやるか・なぜやるか |
| TODO.md | 作業中のフロー管理 | タスク分割・進捗・実装計画 |
| PR | 作業の記録 | 実際にやったこと・計画からの差分 |

### フロー

```
1. Issue を作成する（計画を記録）
2. ブランチを切る: feat/#<issue番号> / fix/#<issue番号>
3. TODO.md で実装を進める（既存の開発フローに従う）
4. 実装完了 → PR を作成する
   - 変更概要
   - 計画からの差分があれば記載
   - Closes #<issue番号>
5. オーナーが確認 → マージ
6. git switch main && git pull origin main
```

### コミットメッセージ

以下の prefix を使う:

- `feat:` 新規追加（ファイル・行）
- `fix:` 修正（挙動や表示の変更を多少なりとも伴う）
- `ref:` リファクタリング（挙動や表示の変更を伴わないコード編集、ファイル移動、命名変更など）
- `test:` テストコード

#### コミットメッセージの書き方

- 最終的な状態のみを記述する。途中経過（「〇〇をやろうとしたが××に変更した」等）は書かない
- ただし以下は必ず記載する:
  - うまくいかなかったこと・妥協した点
  - 別のIssueでの対応とした事項

#### コミットの粒度

機能単位・関心事単位でコミットを分ける。1コミットに複数の関心事を混ぜない。

### やらないこと

- オーナーの承認なしに push / PR作成 / マージをしない
- Issue を勝手にクローズしない
